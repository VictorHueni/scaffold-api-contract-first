MIDDLEWARE REALITY

Why this is tricky on an ESB

01
Manual OpenAPI Import
Most ESBs require a manual import of OpenAPI specs via a file browser or admin console. No native watch or auto-sync from Git. Every spec change means a manual re-import with risk of version mismatch.
02
Scaffolding is Create-Oriented
ESBs can scaffold a new API project from an OpenAPI spec, but the workflow focuses on initial creation. Updating existing flows, mappings, and transformations after a spec change is rarely a supported path.
03
Spec Changes Ripple Through Flows
Renaming a field or changing a response schema can affect message maps, transformation nodes, and connector logic. Flows reference schema fields directly, making the impact hard to predict without good test coverage.
04
OpenAPI Feature Restrictions
Many ESBs don't support the full OpenAPI 3.0 spec. Constructs like oneOf, anyOf, and polymorphism have limited or no support. Your spec may be valid OpenAPI but the middleware can't fully consume it.
05
Flow Development Requires Proprietary Tooling
Message flow development (node wiring, maps, transformations) typically requires a vendor-specific desktop IDE or studio. Harder to integrate into a modern Git-based workflow.
06
CI/CD is Assemble-It-Yourself
Most ESB vendors don't ship an official CI/CD pipeline. Artifact builds rely on community tools or custom scripts. Teams assemble their own pipelines from scratch.
These are real constraints. The demo works within them.
