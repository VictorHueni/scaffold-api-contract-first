YOUR INTEGRATION STACK

Where Contract-First Gets Tricky

01
Manual OpenAPI Import
The gateway admin console requires a manual import of OpenAPI specs — no native Git sync. Every spec change means a manual re-import, with the risk of a version mismatch between what's deployed and what's in the repo.
02
Scaffolding is Create-Oriented
The integration toolkit can scaffold a new project from an OpenAPI spec, but the workflow is designed for initial creation only. Updating existing flows, mappings, and transformations after a spec change is not a supported path.
03
Spec Changes Ripple Through Flows
Renaming a field or changing a response schema can affect message maps, transformation nodes, and connector logic. Flows reference schema fields directly, making the impact hard to predict without good test coverage.
04
OpenAPI Feature Restrictions
The platform does not support the full OpenAPI 3.0 feature set. Constructs like oneOf, anyOf, and polymorphism have limited or no support. A spec that is fully valid OpenAPI may still cause import errors or silent failures.
05
Flow Development Requires Proprietary Tooling
Message flow development — node wiring, maps, transformations — requires a vendor-specific desktop toolkit. This makes it harder to integrate flow authoring into a standard Git-based, PR-driven workflow.
06
CI/CD is Assemble-It-Yourself
There is no official CI/CD pipeline for the integration platform. Artifact builds rely on community tools and custom scripts. Teams assemble their own pipeline from scratch, with limited vendor support.
These are real constraints. The approach works within them.

YOUR DECISIONS

What Belongs Where?

In the OpenAPI Spec
Endpoints, paths & HTTP methods
Request / response schemas & data types
Security schemes (OAuth2 scopes, API keys)
Error codes & error body shapes
Examples — become mock & contract test data
Deprecation & versioning markers
In Confluence
Business context & use cases
Non-functional requirements (performance, SLAs)
CISO security validation checklist
API Style Guide & naming conventions
Governance & approval process
Architecture decisions (ADRs)
Style Guide → Spectral rules: conventions documented in Confluence become linting rules enforced automatically on every push — no manual review gate needed.

Git Branching Strategy for the Contract

A Trunk + Short-lived
Feature Branches Recommended
Small, reviewable diffs per PR
PR is the natural SA→ESB handoff
Branches must stay short-lived
B Long-lived
Design Branch
SA iterates freely, no review friction
Large batch diffs, harder to review
Branch drifts, conflicts accumulate

(develop + features) Multi-env
In-progress vs deployed clearly separated
Scales to Staging + Prod APIC envs
Two PR layers — overhead today
D Separate Contract Repo
+ CI Deploy Future
Contract decoupled from APIC infra
CI→APIC not yet available on IBM
