Why this is tricky on IBM

01
Manual OpenAPI Import
ACE Toolkit requires a manual import of OpenAPI specs via file browser. No native watch or auto-sync from Git. Every spec change means a manual re-import with risk of version mismatch.
02
ACE Scaffolding is Create-Oriented
ACE can scaffold a new REST API from an OpenAPI spec, but the workflow focuses on initial creation. Updating existing flows, subflows, and mappings after a spec change is not a supported path today.
03
Spec Changes Can Ripple Through Flows
Renaming a field or changing a response schema can affect ACE message maps, compute nodes, and connector logic. Flows reference schema fields directly, making the impact hard to predict without good test coverage.
04
OpenAPI 3.0 Feature Restrictions
ACE doesn't support the full OpenAPI 3.0 spec. Constructs like oneOf, anyOf, and polymorphism have documented restrictions. Your spec may be valid OpenAPI but ACE can't fully consume it.
05
Flow Development Requires Desktop Toolkit
ACE v12 added a web editor for API design, but message flow development (node wiring, maps, compute nodes) still requires the Eclipse-based desktop toolkit. Harder to integrate into a modern Git-based workflow.
06
CI/CD for On-Prem ACE is Exploratory
No official IBM CI/CD product for on-prem ACE. BAR builds rely on community tools like ace-maven-plugin and ace-demo-pipeline, provided AS-IS with no warranty. Teams assemble their own pipelines.
These are real constraints. The demo works within them.
