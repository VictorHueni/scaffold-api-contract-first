Why Contract-First?

Code-First vs Contract-First

Same project, same team, same API. Different starting point.

Code-First

Sequential
Week 1
BackBuild API endpoints from scratch
FrontBlocked — waiting on backend
QABlocked — nothing to test
ProdBlocked — no docs to review
Week 2
BackFinish API, deploy to dev
FrontStill blocked — API not stable
QAStill blocked
Week 3
FrontStart integrating — find field name mismatches
BackFix mismatches, break other consumers
QAWrite Postman collection by hand
Week 4
QATest — find integration bugs
ProdWrite docs in Confluence
FrontFix UI against changed API
Week 5-6
BackFix bugs found by QA
QARetest — Postman collection already stale
DevOpsConfigure gateway manually
Week 7
FrontRetest after backend fixes
ProdDocs already stale again
~7 weeks
Sequential handoffs, late bugs, docs drift from day one

Contract-First

Parallel
Day 1
LeadWrite OpenAPI spec, PR reviewed by all roles
Day 2
FrontBuild against Prism mock + generated TS client
BackFill in generated Express stubs
QASchemathesis runs in CI — 1000+ tests, zero hand-written
ProdReview Scalar docs — always current
DevOpsCI pipeline active on every push
Week 1
FrontUI complete — built against typed mock data
BackBusiness logic complete — contract tests pass locally
QAWrite Hurl tests for business scenarios
Week 2
FrontSwitch from mock to real backend — same types, same shape
BackEdge cases and performance
QAAll contract tests + Hurl tests pass
Week 3
LeadIntegration complete — zero surprise mismatches
ProdDocs were never stale — Scalar generated from spec
DevOpsPipeline green — breaking changes blocked by oasdiff
~3 weeks
Everyone starts day one, parallel work, zero drift, contract-tested

Same team. Same API. Same scope.
The only difference is where you start: from code, or from a contract.
